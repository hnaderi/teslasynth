# Getting started

## What you need

- An ESP32, ESP32-S2, or ESP32-S3 development board
- A USB cable (for flashing and power)
- A Chrome or Edge browser (for the web installer)
- Your interruptible device connected to an output GPIO pin

The firmware ships with **4 output channels** configured on some of the safe GPIO pins by
default. You can disable outputs and/or change pin assignments later through the
web dashboard.

#### ⚠ Note on GPIO selection
Before choosing a GPIO pin, ensure that it's not a strapping pin.
Each chip model has its own strapping pins, that are not safe to use for GPIO usage.
Using these pins can cause sending pulses, uncontrolled by the firmware, during boot time,
or other arbitrary times. Do a quick search like `strapping pins esp32s2` before choosing a pin.

On **ESP32-S2**, GPIO 37 and GPIO 38 are reserved for the serial console
(UART TX and RX respectively). Do not assign output channels to these pins.

## Installing the web app

For quick access, you can install the Teslasynth website as a standalone app.
In Chrome or Edge, click **Install app** in the navigation bar when it appears,
or use the browser's install option (address bar icon or menu). The installed
app opens without browser chrome and keeps the MIDI player and console one
click away.

## Installing the firmware

The easiest way to install Teslasynth is directly from your browser — no
software installation required.

1. Plug your ESP32 board into your computer via USB.
2. Open the [Teslasynth website](/) in Chrome or Edge.
3. Click **Install firmware** on the home page.
4. When the browser asks, select the serial port for your board.
5. If prompted, hold the **BOOT** button on your board, press and release
   **RESET**, then release **BOOT**. This puts the board into bootloader mode
   on adapters that don't support auto-reset.
6. Follow the on-screen prompts. The installer will ask whether to erase the
   device — choose **Erase** for a clean first install, skip it for upgrades if
   you want to keep your settings.
7. Wait for the progress bar to complete. The board reboots automatically when
   done.

To install a specific firmware version, use the
[version selector](/flasher) instead.

If the web installer doesn't work for your board or operating system, see
[Troubleshooting](#troubleshooting) at the bottom of this page.

## Connecting to the device

The web dashboard and Wi-Fi access point are only active in **maintenance
mode**. After a clean flash the device boots into maintenance mode automatically
so you can configure it before first use. On subsequent boots, hold the **BOOT**
button for 3 seconds to enter maintenance mode manually (watch for the slow then
fast LED blink — release during the fast blink to confirm).

Once in maintenance mode:

1. On your phone or computer, connect to the Wi-Fi network **Teslasynth**.
   The default password is `Wardenclyffe1891!`
2. Open **http://teslasynth.local** in a browser. If mDNS doesn't work on your
   device, use **http://192.168.4.1** instead.
3. The web dashboard opens. From here you can configure outputs, MIDI routing,
   and hardware settings.

See [Configuration](/configuration) for the full list of settings and CLI
commands.

## ⚠ Before connecting high-voltage hardware

**Please read the [Configuration](/configuration) page carefully before any
high-voltage tests.** In particular, the duty cycle default is 100% — safe for
passive loads and simulations, but must be set to a safe value for your specific
device before energising it. Connecting a DRSSTC or flyback transformer without
configuring these limits first risks destroying your hardware or causing injury.

## Wiring an output

Connect your output device's interrupter input to the GPIO pin configured for
channel 1 and to **GND**. The signal is a 3.3 V logic
pulse — your interrupter circuit must accept this level or use a level shifter.

The output is **active-high**: the pin goes high during a pulse and returns low
between pulses.

**Do not connect high voltage directly to the ESP32.** Always use an optocoupler
or a dedicated interrupter board as the interface between the ESP32 and your
high-voltage device.

Add a **pull-down resistor** (10 kΩ is typical) from each output GPIO to GND.
Without it, the pin floats during boot and reset, which can send spurious pulses
to your device before the firmware takes control of the pin.

## Confirming it works

Open the [**Tools**](/tools) page on the website and expand the **Console**
panel. Click **Connect**, select your board's serial port, and you should see
boot messages from the firmware — firmware version, chip info, and a command
prompt waiting for input.

Once your device is playing MIDI, you can also use the **MIDI Player** panel on
the same page to load a `.mid` file and play it directly from the browser,
with the serial console open alongside it.

## Troubleshooting

### ESP32-S2 boards with native USB (S2-Mini, S2-Saola)

Boards where the chip's USB-OTG pins are wired directly to the USB connector
(no USB-to-UART bridge chip) **cannot be flashed through the web installer**.
The web installer relies on toggling the DTR/RTS control lines to put the chip
into download mode, but USB-OTG CDC doesn't expose those signals — the call
fails with `Failed to set control signals` and the flash aborts before sync.

`esptool.py` works around this with a magic-baud-rate reset that triggers the
chip's USB stack to reboot into download mode. The JavaScript port of esptool
hasn't implemented that path yet, so for now use the command line:

```sh
idf.py -p /dev/ttyACM0 flash
```

Or wire a USB-to-UART adapter to the chip's UART0 pins and use that port with
the web installer instead.

### Linux: ModemManager interferes with the serial port

On most Linux distributions, **ModemManager** automatically probes any new
USB-CDC device looking for AT-command modems. When you plug in an ESP32, it
sends junk into the serial port for several seconds — long enough to break
flashing and corrupt console output. Symptoms include:

- The web installer times out at "Connecting…"
- Random bytes appearing in the serial console after plug-in
- `idf.py flash` failing the first time but succeeding on retry

The cleanest fix is to tell ModemManager to ignore Espressif's USB IDs:

```sh
sudo nano /etc/udev/rules.d/99-esp32.rules
```

```
# Tell ModemManager not to probe Espressif USB devices
SUBSYSTEM=="tty", ATTRS{idVendor}=="303a", ENV{ID_MM_DEVICE_IGNORE}="1"
SUBSYSTEM=="tty", ATTRS{idVendor}=="10c4", ENV{ID_MM_DEVICE_IGNORE}="1"
SUBSYSTEM=="tty", ATTRS{idVendor}=="1a86", ENV{ID_MM_DEVICE_IGNORE}="1"
```

Then reload:

```sh
sudo udevadm control --reload-rules && sudo udevadm trigger
```

If you don't use a modem at all, the simpler option is to disable
ModemManager entirely:

```sh
sudo systemctl disable --now ModemManager
```
