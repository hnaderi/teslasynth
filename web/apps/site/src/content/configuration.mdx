# Configuration

Teslasynth stores its configuration in NVS (non-volatile storage) on the device
flash. Settings survive reboots and firmware upgrades, unless you explicitly
erase or reset them.

There are two ways to configure the device:

- **Web dashboard** — boot into maintenance mode (see below), connect to the
  device's Wi-Fi AP, and open http://teslasynth.local.
- **Serial CLI** — connect via the serial console and use the `config` command.
  More powerful: supports wildcards, bulk updates, and scripting.

> **ESP32-S2 note:** On the S2, the serial console is only available over UART
> (GPIO 37 TX / GPIO 38 RX, 115200 baud), not over the native USB port. This is
> because the ESP-IDF USB CDC console relies on the ROM CDC stack, which cannot
> coexist with the TinyUSB stack that Teslasynth uses for USB MIDI. The ESP32-S3
> is not affected — it has a dedicated USB Serial/JTAG peripheral that is
> independent of the USB OTG port used for MIDI.

## Maintenance mode

The web dashboard and Wi-Fi AP are only active in **maintenance mode**. During
normal operation the device runs the synthesizer with no network stack, keeping
all resources available for audio.

There are two ways to enter maintenance mode:

**Physical button** — hold the maintenance button (default: **GPIO 0**, the
BOOT button on most dev boards, or alternativly, ground the **GPIO 0**) and watch the status LED:

| LED | Meaning |
|-----|---------|
| Slow blink | Hold detected — keep holding |
| Fast blink | 3 seconds reached — release now |
| Solid | Confirmed — rebooting into maintenance mode |
| Off | Released too early — aborted, try again |

Release the button during the fast-blink phase to confirm. If you release too
early the sequence aborts and the LED goes off.

The LED indicators require a status LED to be configured in the hardware
settings. Without one, the sequence still works — there is just no visual
feedback.

**First install** — if no valid hardware configuration is stored (e.g. after a
clean flash), the device automatically boots into maintenance mode so you can
configure it before first use.

**CLI command** — from the serial console, run:

```
maintenance
```

Once in maintenance mode:
1. Connect to the Wi-Fi network **Teslasynth** (default password:
   `Wardenclyffe1891!`).
2. Open **http://teslasynth.local** (or **http://192.168.4.1** if mDNS doesn't
   work on your device).
3. Configure the device using the web dashboard.
4. Reboot (via the dashboard or the `restart` CLI command) to return to normal
   synthesis mode.

## CLI config command

```
config [<key[:<ch>]=value> ...] [flags]
```

Called with no arguments it prints the full current configuration. Pass one or
more `key=value` pairs to update settings.

**Flags:**

| Flag | Description |
|------|-------------|
| `-s`, `--save` | Persist changes to NVS flash |
| `-r`, `--reload` | Reload synthesizer with new config immediately |
| `--reset` | Reset all config to firmware defaults |

Changes are **not** saved automatically — always add `-s` if you want them to
survive a reboot. Add `-r` to hear the effect immediately without rebooting.

**Examples:**

```
config
config synth.tuning=432hz
config output.1.max-duty=15.5 output.2.notes=2 --save
config output.*.max-on-time=200us -r -s
config --reset
```

## Global synth settings

| Key | Format | Default | Description |
|-----|--------|---------|-------------|
| `synth.tuning` | `<n>[hz\|khz]` | `440hz` | Reference tuning frequency |
| `synth.instrument` | `<1–28>` or `-` | — | Default instrument for all outputs |

**Frequency format:** `440`, `440hz`, or `0.44khz` are all accepted.

`synth.instrument` sets a global fallback used by any output that does not have
its own instrument assigned. Set it to `-` to clear it.

## Per-output settings

Replace `<ch>` with an output number (`1`–`4`) or use `*` to apply a value to
all outputs at once.

| Key | Format | Default | Description |
|-----|--------|---------|-------------|
| `output.<ch>.max-on-time` | `<n>[us\|ms]` | `100us` | Maximum continuous on-time per pulse |
| `output.<ch>.min-deadtime` | `<n>[us\|ms]` | `100us` | Minimum dead time between pulses |
| `output.<ch>.max-duty` | `0–100` (float) | `100` | Maximum duty cycle (%) |
| `output.<ch>.duty-window` | `<n>[us\|ms]` | `10ms` | Time window for duty cycle enforcement |
| `output.<ch>.pulse-resolution` | `<n>[us\|ms]` | `0us` | Smallest on-time the bridge can actually deliver (`0` disables correction) |
| `output.<ch>.notes` | `1–4` | `4` | Maximum concurrent notes |
| `output.<ch>.instrument` | `<1–28>` or `-` | — | Instrument override for this output |

**Duration format:** `100`, `100us`, or `10ms`. Maximum value is 65535 µs.

**Duty cycle** is the fraction of time the output is on within the duty window.
It is the primary safety limit for your device.

The default of **100%** is intentional: it imposes no limit, which is safe for
passive loads (speakers, LEDs) and Python simulations. **Before connecting any
high-voltage device, set this to a value appropriate for your hardware:**

- **DRSSTCs** — typically 5–10%; the resonant tank stores energy and even short
  bursts carry significant power
- **Flyback transformers** — up to ~50% is often acceptable depending on the
  core and drive circuit
- **Other interrupters** — consult your device's documentation; when in doubt,
  start at 5% and increase gradually while monitoring temperature

**Max on-time** caps the length of any single pulse regardless of the note being
played.

**Min deadtime** ensures a minimum off-time between pulses. For DRSSTCs this is
critical: the tank circuit current must ring down to zero before the next burst,
or the bridge switches under load — set it conservatively until you have
confirmed safe operation on your specific coil.

**Pulse resolution** corrects the duty limiter for coils that can't actually
deliver the on-time the firmware requests.

A DRSSTC bridge fires with zero-current switching: once triggered, it has to
complete at least one half-cycle of the resonant tank before it can turn off.
So a 1µs trigger sent to a 100 kHz coil produces a real 5µs burst — five times
longer than asked for. Without telling the firmware about this, the duty limiter
charges only the 1µs it requested, undercounts what the coil actually does, and
a stream of tiny pulses can quietly drive the real duty cycle several times past
your configured `max-duty` before anything intervenes. That is the kind of
mismatch that destroys IGBTs.

Setting `pulse-resolution` to **half the resonant period** of the coil tells the
firmware to charge the duty limiter for what the bridge really delivers, and to
extend the off-time so `min-deadtime` is still measured against the real
switch-off — not the firmware's request.

| Resonant frequency | `T/2` | `pulse-resolution` |
|---|---|---|
| 100 kHz | 5.00 µs | `5us` |
| 150 kHz | 3.33 µs | `4us` |
| 200 kHz | 2.50 µs | `3us` |
| 300 kHz | 1.67 µs | `2us` |

The general rule: `pulse-resolution = 1 / (2 × fres)`, rounded **up** to the
nearest microsecond. Rounding up keeps the duty limiter conservative — it
will charge for slightly more than the bridge actually delivers, never less.

Leave at `0` for hard-switched outputs — LEDs, speakers, simulation, or any
non-resonant interrupter — where the firmware's requested on-time matches the
delivered on-time exactly.

## MIDI routing

| Key | Format | Description |
|-----|--------|-------------|
| `routing.channel.<midi_ch>` | `<1–4>` or `-` | Map MIDI channel (1–16) to an output, or `-` to unroute |
| `routing.percussion` | `y` / `n` | Enable MIDI channel 10 percussion routing |

Use `*` as the MIDI channel to set all 16 channels at once.

```
config routing.channel.*=-        # unroute all channels
config routing.channel.1=1        # route MIDI ch 1 to output 1
config routing.channel.2=1        # also route MIDI ch 2 to output 1
config routing.channel.3=2 -s     # route MIDI ch 3 to output 2, save
config routing.percussion=y -s    # enable percussion
```

## Hardware configuration

Hardware settings (GPIO pins, LED) are configured through the web dashboard
under **Hardware**. They require a reboot to take effect and are stored
separately from the synth configuration.

Use the `hwconfig` CLI command to inspect the current hardware configuration:

```
hwconfig
```

To show compile-time limits (max notes, number of outputs):

```
limits
```
